## Tabla de Pruebas вЂ” Column Conventions

| Column | Type | Rules |
|--------|------|-------|
| N | int | Sequential 1,2,3... Used in trazabilidad matrix |
| TC-ID | string | `TC-{MODULE}-{SUB}-{NNN}` from specs; or `TC-{MODULE}-{NNN}` if no submodule code |
| Suite / ГЃrea Funcional | string | Matches a suite name from section 5 |
| TГ­tulo | string | `[ETIQUETA][P#] Description` |
| DescripciГіn | string | What scenario this validates (1-2 sentences) |
| Steps | markdown | Numbered, `<br>`-separated. Concrete UI verbs. |
| Resultado Esperado | string | Observable, measurable outcome |
| Confirma | string | `Task NNNNN` or `N/A` |
| Tipo | enum | `Manual` \| `Automatizado` \| `Ambos` \| `Bloqueado` |
| Prioridad | enum | `P0` \| `P1` \| `P2` \| `P3` |

### Title Etiquetas
- `[CONFIRMACION]` вЂ” confirms a specific sprint item is implemented
- `[SMOKE]` вЂ” happy path end-to-end
- `[REGRESION]` вЂ” validates no regression in adjacent functionality
- `[EXPLORATORIA]` вЂ” charter-based, timeboxed, no fixed expected result

## Steps Writing Rules

**DO:**
- `1) Navegar a {QA_BASE_URL}/ruta/modulo`
- `2) Ingresar usuario {QA_USER_EMAIL} y contraseГ±a {QA_USER_PASSWORD}`
- `3) Hacer click en el botГіn "Guardar"`
- `4) Verificar que aparece el mensaje "Registro guardado correctamente"`
- `5) Revisar bandeja de correo del usuario {QA_NOTIFY_EMAIL}`

**DO NOT:**
- `1) Ejecutar la funcionalidad` в†ђ too vague
- `2) Verificar que funciona` в†ђ not measurable
- `3) Hacer las acciones necesarias` в†ђ not observable

**References to test data:**
- Env vars: `{QA_USER_EMAIL}`, `{QA_ADMIN_EMAIL}`, `{QA_BASE_URL}`
- Dynamic values: `EXEC_IDX` (unique run suffix)
- Specific values: write the actual value if it's fixed (e.g., a range of dates like `01-03-2026 a 07-03-2026`)

**Exploratory charters:**
```
1) Timebox: 45вЂ“60 min
2) Objetivo: {CHARTER_GOAL}
3) Explorar {AREA} variando {VARIABLES}
4) Registrar hallazgos con evidencia (screenshot + descripciГіn)
```

## Priority Decision Tree

```
Is it the primary happy path?                    в†’ P0
Is it a show-stopper risk or data corruption?    в†’ P0
Is it a common negative scenario?                в†’ P1
Is it a cross-module dependency?                 в†’ P1
Is it a state machine transition?                в†’ P1
Is it a secondary feature (export/search)?       в†’ P2
Is it regression in adjacent functionality?      в†’ P2
Is it an edge case or low-frequency scenario?    в†’ P3
Is it exploratory / cosmetic?                    в†’ P3
```

## Automation Feasibility Rules

| Feasibility | Criteria | Column `Tipo` |
|---|---|---|
| Fully automatable | Deterministic, observable via browser, no external deps | `Automatizado` |
| Partially automatable | Some steps require human inspection OR external can be mocked | `Ambos` |
| Not automatable | Physical access; irreversible QA side effects; BLOCKED-PERMISSIONS | `Manual` |
| Blocked | Feature not deployed or BLOCKED-PERMISSIONS | `Bloqueado` |

## Multi-Module Sprint

When a sprint covers multiple modules:
- Create one `Plan-de-Pruebas-{proyecto}-Sprint-{N}-{modulo}.md` per module
- The consolidated `Plan-de-Pruebas-{proyecto}-Sprint-{N}.md` is generated by `@keber/ado-qa consolidate-sprint`
- **Do not create the consolidated file manually** вЂ” it is a derived artifact
